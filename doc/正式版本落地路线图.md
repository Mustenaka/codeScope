# codeScope 正式版本落地路线图
> 更新日期：2026-03-19
> 适用范围：`bridge` / `server` / `mobile` / `doc`
> 基线：`project -> thread -> message -> thread_state -> file browser -> p2p direct / relay fallback`

## 1. 需求理解

目标场景不是“远程看事件流”，而是：

- 用户短时间离开电脑时，能在手机上实时看到哪些项目正在活跃
- 每个项目里有哪些对话线程正在运行、等待用户、已完成或已阻塞
- 每个线程里真实收发了什么内容
- 用户可以在手机上继续发送 prompt，在线程内接续修改
- 消息正文、线程历史、文件内容尽量走 `bridge <-> mobile` 直连，`server` 默认只做发现、路由和可选 relay fallback

## 2. 当前现状

### 已实现

- `server` 已提供 `project/thread/message` 只读 API，同时保留 legacy `sessions` 兼容接口
- `mobile` 主视图已经切到 `project -> thread -> message`
- `thread detail` 已支持基础消息历史、摘要和 `thread_state`
- `project/thread/detail` 已支持基础实时刷新
- `bridge` 已带基础 `project_name / workspace_root / thread_id / thread_state` 语义
- `bridge` 已开始旁路读取 Codex 本地 `session_index.jsonl` 与 `sessions/*.jsonl`，可采到一部分真实 `user / assistant` 文本

### 半实现

- `thread` 仍带明显的 `session` 派生痕迹，聚合规则仍是兼容层思路
- `thread_state` 仍有较多启发式推导，不是严格生命周期状态机
- 真实消息采集目前偏向 Codex 本地 session 文件，尚未形成统一 capture adapter 抽象
- 列表实时更新仍大量依赖“事件触发后重拉 REST 快照”
- `mobile -> server -> bridge -> agent` 的正式 prompt 闭环尚未落成

### 未实现

- 完整真实输入/输出历史采集
- 项目级 `create thread` 主链路
- `bridge <-> mobile` 默认 P2P 直连与 relay fallback 正式协议
- 面向 release 的精确 `thread_state` 生命周期闭环
- 关键状态变化通知：`completed / failed / awaiting_user / quiet-period new reply`

## 4. 修改方案

### Phase 1：稳定 `project/thread/message` 兼容主链路

- 继续保留 legacy `session/event`
- 把 `server` 的 release 读模型彻底收敛到 `project/thread/message`
- 明确 `thread_id`、`source_session_id`、`thread_title` 的聚合规则
- 让空白 debug-only thread 不再进入 release 主视图

### Phase 2：补齐真实消息采集

- 为 `Codex` 建立稳定的 session file capture adapter
- 为 `Claude Code` 设计对等的 message capture adapter
- 把 `message` 采集、线程归属、状态推导从 discovery/debug 层拆开
- 优先采真实 `user / assistant / tool_call / tool_result`

### Phase 3：建立正式 `thread_state` 状态机

- 把当前启发式推导替换为事件驱动状态机
- 最低要支持：
  - `running`
  - `waiting_prompt`
  - `waiting_review`
  - `completed`
  - `blocked`
  - `offline`
  - `stale`

### Phase 4：打通手机继续发 prompt 的正式闭环

- `mobile` 支持项目级新建 thread
- `mobile` 支持向已有 `thread` 发送 prompt
- `server` 负责路由、授权、投递状态
- `bridge` 负责把 prompt 投递到本地 agent，并把后续真实消息回流

### Phase 5：P2P 直连与 relay fallback

- `server` 默认变成 discovery / route broker
- `bridge <-> mobile` 默认直连传正文和文件内容
- 直连失败时自动退到 relay fallback
- 明确内容层和索引层分离

### Phase 6：项目级 file browser 完整化

- 文件树和文件内容完全按 `project` 语义暴露
- 明确只读预览边界
- 与 thread 详情并列成为项目主视图组成部分

## 6. 风险与回归点

- `Codex` 和 `Claude Code` 的真实消息来源并不一致，capture adapter 容易走成两套临时逻辑
- 继续沿用 `session` 做核心主键，会反复把一个真实 thread 拆成多个兼容 thread
- 如果 `thread_state` 继续只看进程/心跳，会持续误判 `running`
- 如果 `mobile` 继续靠 REST 全量重拉，弱网和断线恢复体验会很差
- 如果 `server` 继续默认保存正文，会和 P2P 隐私目标冲突
- 文档如果继续保留重复版本，很容易让后续代理重新走回 session/event 视角

## 7. 建议实施顺序

1. 先固定文档基线和提示词，删掉重复且误导的旧文档。
2. 再把 `bridge/server` 的 thread 聚合和真实 message 采集补稳定。
3. 再把 `thread_state` 收敛成正式状态机。
4. 再打通 `mobile -> prompt -> bridge -> agent -> message 回流` 的闭环。
5. 最后推进 `p2p direct / relay fallback` 与项目级 file browser 完整体验。

## 一句话结论

正式版本的关键不是继续“把事件看得更清楚”，而是把当前兼容层逐步收敛成一个真正面向 `project / thread / message / thread_state` 的移动端远程协作系统。
